# Production Deployment Guide

## Purpose

Use this document as the production release gate and operating baseline for `cam-platform`.

This guide is intentionally stricter than the general deployment guide. It focuses on:

- what must be true before an internet-facing rollout
- how the production stack is expected to be configured
- hardening steps that are mandatory, not optional
- validation, backup, rollback, and acceptance criteria

Pair it with:

- [Deployment Guide](./DEPLOYMENT_GUIDE.md)
- [Commissioning Checklist](./COMMISSIONING_CHECKLIST.md)
- [Operations Guide](./OPERATIONS_GUIDE.md)
- [Incident Runbook](./RUNBOOK_INCIDENTS.md)

## Executive Summary

The control plane deployment in [`deploy/docker-compose.yml`](../deploy/docker-compose.yml) is now much closer to a production shape. Bootstrap builds installer artifacts, the API serves release binaries, and provisioning attempts server-side WireGuard peer registration. Treat this as a controlled rollout guide, not an unconditional public launch approval.

Current blockers:

1. Live stream proxying still needs final field validation.
   The browser, API, WireGuard path, Frigate, and a real on-prem camera need to be verified together in a production-like environment.

Production launch status:

- `Allowed`: internal testing, staging, pilot, controlled customer rollout with manual operator involvement
- `Not allowed yet`: unattended large-scale production rollout or broad public launch until live-stream validation is complete

## Production Readiness Gates

All of the following must be complete before calling the platform production-ready.

### Gate 1: Functional readiness

- one full edge install succeeds without manual patching of the installer
- dashboard login, camera list, snapshots, and live view work against a real remote site
- at least one rollback rehearsal has been completed

### Gate 2: Security readiness

- DNS for `app`, `api`, `auth`, and `metrics` points to the production host
- TLS cert issuance succeeds cleanly through Caddy
- Grafana access is restricted in [`Caddyfile`](../deploy/caddy/Caddyfile)
- all bootstrap-generated secrets are stored in a vault
- SSH is restricted to admin IP ranges
- backup and restore procedures have been tested

### Gate 3: Operational readiness

- API, Keycloak, Postgres, and Caddy health checks are green
- host disk, memory, and container restart alerts are configured
- Postgres backup schedule is active
- on-call responders know where the runbook and credentials live
- a first-customer commissioning checklist has been rehearsed

If any gate is incomplete, the platform is not production-ready.

## Infrastructure Requirements

### Control plane host

- Ubuntu 24.04 LTS
- static public IP
- recommended minimum:
  - `4 vCPU`
  - `8 GB RAM`
  - `80 GB SSD`
- inbound ports open:
  - `80/tcp`
  - `443/tcp`
  - `443/udp`
  - `51820/udp`
- Docker Engine and Docker Compose plugin
- WireGuard kernel support

### Edge host

- Ubuntu 22.04+ or Debian-compatible Linux
- recommended minimum:
  - `4 CPU cores`
  - `8 GB RAM`
  - storage sized for recording retention
- Docker support
- WireGuard support
- LAN reachability to the camera subnet
- outbound access to:
  - `https://api.<domain>`
  - `https://auth.<domain>`
  - `udp/51820` on the control plane host

### DNS

Create A records pointing to the production host:

- `app.<domain>`
- `api.<domain>`
- `auth.<domain>`
- `metrics.<domain>`

Do not start the public stack until these resolve correctly.

## Stack Configuration Baseline

The production compose file is [`deploy/docker-compose.yml`](../deploy/docker-compose.yml).

### Required environment values

These values are generated by [`setup.sh`](../deploy/setup.sh) into `deploy/.env`:

- `DOMAIN`
- `POSTGRES_PASSWORD`
- `KC_ADMIN_USER`
- `KC_ADMIN_PASSWORD`
- `MQTT_API_USER`
- `MQTT_API_PASSWORD`
- `DEVICE_SECRET`
- `GRAFANA_PASSWORD`
- `NEXTAUTH_SECRET`
- `CAM_DASHBOARD_SECRET`
- `CAM_API_SECRET`
- `GRAFANA_KC_SECRET`
- `WG_SERVER_PUBLIC_KEY`
- `METRICS_ALLOWLIST`

Optional archive settings:

- `S3_ENDPOINT`
- `S3_BUCKET`
- `S3_ACCESS_KEY`
- `S3_SECRET_KEY`

### Identity and issuer requirements

The API and dashboard intentionally separate internal Keycloak access from the public issuer:

- `KEYCLOAK_URL=http://keycloak:8080`
- `KEYCLOAK_ISSUER=https://auth.<domain>/realms/camplatform`

That split must remain in production:

- internal service-to-service traffic stays on the Docker network
- token issuer validation matches public browser-visible URLs

### Reverse proxy requirements

[`Caddyfile`](../deploy/caddy/Caddyfile) already defines:

- TLS termination for `app`, `api`, `auth`, and `metrics`
- HSTS headers
- basic hardening headers
- HTTP/3 on `443/udp`

Before launch, additionally:

- restrict `metrics.<domain>` to office IPs, VPN, or an allowlist
  - update `METRICS_ALLOWLIST` in `deploy/.env` (CIDR list, e.g. `203.0.113.10/32 198.51.100.0/24`)
- decide whether Caddy rate limiting will be enabled for the API
- verify no other host process is binding `80` or `443`

## Deployment Procedure

### 1. Prepare the host

```bash
sudo apt-get update
sudo apt-get upgrade -y
sudo timedatectl set-timezone UTC
sudo apt-get install -y git curl
```

Recommended host-level setup before bootstrap:

- create a non-root admin user
- disable password SSH authentication
- enable automatic security updates
- confirm the hostname and time sync are correct

### 2. Clone and bootstrap

```bash
git clone <your-repo-url> cam-platform
cd cam-platform
chmod +x deploy/setup.sh
sudo ./deploy/setup.sh yourdomain.com
```

[`setup.sh`](../deploy/setup.sh) currently:

- installs Docker and WireGuard tools
- writes `deploy/.env`
- creates the Mosquitto password file
- generates WireGuard server keys under `deploy/wireguard`
- builds agent release binaries under `deploy/releases`
- renders `deploy/keycloak/realm.resolved.json`
- runs `docker compose up -d --build`

### 3. Verify service health

```bash
cd deploy
docker compose ps
curl -fs http://localhost:3001/health
docker logs cam_api --tail 100
docker logs cam_keycloak --tail 100
docker logs cam_caddy --tail 100
```

Expected minimum state:

- `cam_postgres` healthy
- `cam_keycloak` healthy
- `cam_mosquitto` healthy
- `cam_api` healthy
- `cam_dashboard` running
- `cam_caddy` running

### 4. Verify public paths

Check:

- `https://app.<domain>`
- `https://auth.<domain>`
- `https://api.<domain>/health`

Expected:

- dashboard loads over TLS
- Keycloak login page loads
- API health responds successfully
- TLS certificates are valid and browser-trusted

### 5. Record bootstrap outputs

Store securely:

- `deploy/.env`
- Keycloak admin password
- Grafana admin password
- `deploy/wireguard/server_private.key`
- `deploy/wireguard/server_public.key`

Do not leave these only on the server filesystem.

## Mandatory Hardening

These steps are required before any real customer traffic.

### Network and host hardening

- allow only `80/tcp`, `443/tcp`, `443/udp`, and `51820/udp` publicly
- restrict SSH to admin IP ranges
- enable a host firewall and cloud firewall together if available
- disable unused host services
- ensure host disk encryption or cloud volume encryption is enabled where supported

### Secret management

- move all generated secrets from `deploy/.env` into a managed secret store
- rotate bootstrap secrets if staging and production ever shared values
- rotate Keycloak admin credentials after first login
- document secret ownership and rotation cadence

### Application hardening

- restrict `metrics.<domain>` access in [`Caddyfile`](../deploy/caddy/Caddyfile)
- review Keycloak realm settings, redirect URIs, and client secrets
- confirm dashboard and API are using only the public issuer URL for token validation
- keep dashboard media proxy endpoints authenticated
- do not ship localhost-only auth bypasses into production

### Dependency and image controls

- pin container image versions before launch instead of relying on floating tags such as `latest`
- capture the exact git commit and compose file used for each deployment
- use a defined upgrade window and rollback point for stack changes

## Backup and Restore Requirements

Minimum backup scope:

- Postgres data
- `deploy/.env`
- WireGuard keys
- Caddy data and config volumes
- Keycloak realm configuration and any persistent identity data

Minimum operational standard:

- nightly Postgres backups
- off-host encrypted backup retention
- documented restore procedure
- restore test performed at least once before customer go-live

A production environment without tested restore capability is not ready.

Reference scripts (included in repo):

- `deploy/backup/backup.sh` — Postgres + config snapshot
- `deploy/backup/restore.sh` — restore a Postgres dump

Suggested cron (nightly at 02:00 UTC):

```bash
0 2 * * * /bin/bash /opt/cam-platform/deploy/backup/backup.sh >> /var/log/cam-backup.log 2>&1
```

## Monitoring and Alerting

At minimum, alert on:

- API health failures
- Keycloak health failures
- Postgres health failures
- container restart loops
- host disk usage
- certificate renewal failures
- WireGuard tunnel count anomalies
- edge heartbeat drop-offs

Recommended operator views:

- current site/device heartbeat status
- recent provisioning attempts

## Edge ANPR (Open Source)

Edge ANPR runs locally as a sidecar on the edge node. It subscribes to Frigate event snapshots, performs OCR, and publishes plate text via MQTT.

Defaults:

- OCR engine: Tesseract
- Language: `eng` (adjustable)
- Topic: `anpr/<camera_name>`

Configuration is set in the edge install script (`deploy/install-agent.sh`) and can be tuned per site:

- `ANPR_LANG`
- `ANPR_REGEX`

Notes:

- Prometheus alert rules live in `deploy/prometheus/alerts.yml`.
- Alert delivery requires a separate Alertmanager or external service integration.
- container health and restart counts
- disk consumption on control plane and edge hosts

## Edge Deployment Policy

Edge deployment now depends on two concrete behaviors in this repository:

- `deploy/setup.sh` builds release binaries into `deploy/releases`
- the API serves those binaries from `/releases/...`
- provisioning attempts to register the peer on the running `cam_wireguard` container

Allowed current use:

- operator-assisted pilots
- staging validation
- controlled demos with manual verification

Before enabling unattended field rollout, complete final live-stream validation and confirm WireGuard peer registration behavior in the target runtime environment.

## Production Validation Checklist

Run this before declaring a release candidate fit for production:

1. Bootstrap a clean control plane host from scratch.
2. Verify login through Keycloak using the public URLs.
3. Create a provisioning token.
4. Install one edge device using the real installer flow.
5. Confirm the device appears in the dashboard.
6. Confirm `wg show` reports an active tunnel on both sides.
7. Discover at least one physical ONVIF camera.
8. Validate snapshot loading in the dashboard.
9. Validate live stream playback through the API proxy.
10. Restart `api`, `dashboard`, and `caddy` and confirm recovery.
11. Restore a Postgres backup into a test environment and validate login plus site metadata.

Do not skip the live-stream validation step. It is one of the known readiness gaps.

## Rollback Standard

Every production change must have:

- a known-good git commit
- a copy of the last-good `deploy/.env`
- a record of image tags or image digests used in the prior release
- a tested operator command sequence for reverting

Minimum rollback process:

1. stop the rollout
2. restore the previous application version or image set
3. restart affected services
4. verify:
   - `https://api.<domain>/health`
   - dashboard login
   - camera list
   - device heartbeats
5. document incident start time, rollback time, and customer impact

## Launch Decision

You can call this stack production-ready only when:

- the remaining live-stream validation blocker is closed
- hardening steps are complete
- backups and restore are tested
- one full pilot site has passed commissioning and live-view validation
- on-call support is prepared for day-2 operation

Until then, the correct status is:

`Production-capable architecture with controlled-rollout limitations, not full production release approval.`
